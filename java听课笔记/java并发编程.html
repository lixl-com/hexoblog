并发编程

一、线程基本概念

1.基本概念

1.1 进程与线程

进程是os分配资源的基本单位（占用内存）；

线程是cpu调度的基本单位，依附于进程。

所谓进程就是线程的容器，线程利用进程中的一些资源，处理代码及指令。

进程与线程的区别：

.根本不同：进程是os分配资源的基本单位，线程是cput调度的基本单位。

.资源：线程共享同一个进程的资源，同时拥有自己的独立资源(如栈)

1.2 多线程

1.什么是多线程：

单个进程中同时运行多个线程，提高cpu的利用率；

避免网络或磁盘IO的等待操作，让cput去调度其他线程；

2.多线程的局限：

.线程过多时，cput切换线程上下文是，会造成额外开销

.任务拆分依赖业务场景无法拆分的时候，就不能使用多线程

.线程安全：争用资源导致数据不一致，或者死锁

2.线程的创建

1.继承Thread类重新run方法
2.实现Runnable接口 重新run方法
3.实现Callable重写call方法

3.线程的使用

1.线程的状态

线程的常用方法

获取当前线程 Thread t = Thread.currentThread();

setName()设置线程名称
setPriority()设置优先级(1-10, 默认为5)

yield() 让出时间片，线程回到runnable状态

sleep(timeMillis) 让线程回到time_wating状态(会抛出Interrupted异常)

join(timeMillis) 当前线程进入time_waiting状态(让出时间片，让其他线程先执行)

守护线程：在所有非守护线程结束的时候，再执行守护线程

线程等待与唤醒：锁、锁池(blocked)、等待池(wating)

wait()– synchronized块内/notify() 唤醒一个/notifyAll()唤醒多个

结束线程：

.stop()方法–不推荐

.共享变量 volatile保证线程可见性

.interrupt 可以打断waiting/time_waiting状态的线程

二、并发编程的三大特性

1.原子性

  1. 什么是并发编程的原子性
    JMM (Java Memory Model) 屏蔽不同硬件和os在内存操作上的差异性，做到跨平台的目的。
    jmm规定：所以变量都会存在主内存里，进行操作的时候需要从内存复制一份到cpu-cache，在线程内部做计算，然后将计算结果写回内存。
    原子性的定义：一个操作是不可分割、不可中断的，当一个线程执行的时候，另一个线程不会影响到。
    (i++操作不是原子的，通过javap -v x.class可以查看jsm代码是否是一步操作)
    多线程操作临界资源时，预期结果要与最终结果一致。

  1. 如何保证并发编程的原子性？

    synchronized关键字： 通过加锁
CAS： 只能保证变量的原子性、ABA问题(加版本AtomicReference)、自旋时间过长(指定次数、longAddr暂存结果)

Lock锁：

ThreadLocal：不再操作临界资源(jdbc connect对象)![0764d92d-f9c2-4b4b-b280-ad482699424c](file:///C:/Users/lixl/Pictures/Typedown/0764d92d-f9c2-4b4b-b280-ad482699424c.png)

2.可见性

cpu执行速度非常快，因此出现了三级缓存；计算的时候，会将数据从内存拿到cpu的本地缓存再进行操作，但其他线程是看不到修改后的值。

volatile ：用来修饰成员变量的关键字、被修饰的变量必须从内存获取(不允许使用缓存)

final、

synchronized：获取锁之后，涉及到的变量会从cpu-cache中删除，然后从内存中重新拿数据，在释放锁后将cache值同步会内存。

Lock：Lock锁本身有state是volatile修饰的，可同步修所有的状态、

3.有序性

cpu乱序执行 (单例DCL new Object 开辟空间、初始化、分配地址)

as-if-serial: CPU级别

happens-before: jvm级别

使用volatile解决指令重排的问题(DCL)

voaltile基于内存屏障，将内存屏障看出一条指令

三、锁

1.锁的分类

  1. 可重入锁、不可重入锁
    synchronized/ReentrantLock/ReentrantReadWritLock都是可重入锁
    ThreadPool.Worker是不可重入锁

  2. 乐观锁与悲观锁
    synchronized/ReentrantLock/ReentrantReadWritLock都是悲观锁；CAS是乐观锁的一种实现。
    悲观锁：获取不到锁资源是会将当前线程挂起(blocker/wating)，调度需要内核参与，比较消耗资源。
    乐观锁：获取不到锁资源时，可以再次让cpu调度，再次尝试获取锁资源。

  3. 公平锁与非公平锁
    synchronized是非公平锁；ReentrantLock/ReentrantReadWritLock提供两种方式。
    有锁的情况下，先排队的是公平锁；先尝试竞争锁，没有拿到的时候再排队

  4. 互斥锁与共享锁
    同一个时间点，只有一个线程持有当前互斥锁；
    同一个时间点，当前共享锁可以被多个线程持有。

2.深入synchronized

  1. 类锁、对象锁
    静态方法、静态代码库 是类锁
    方法或普通代码块 是对象锁

  2. synchronized优化
    jdk1.6进行的大量优化
    锁消除：没有操作临界资源，就会消除该锁
    锁膨胀：一个循环中，频繁获取与释放锁会导致消耗比较大，锁会向循环体外扩一层。
    锁升级：无锁–>偏向锁–>轻量级锁(CAS)–>重量级锁

  3. 实现原理
    synchronized基于java对象实现的。
    JMM 对象(8的倍数)：对象头(markword 、classpointer)、实例数据、对象填充

  4. 锁升级
    使用openjdk-jol可以看到java对象的markword信息。
    锁默认情况下，开启了偏向锁延迟（5s）；

  5. 重量级锁ObjectMonitor
    jdk8u/jdk8u/hotspot: log (openjdk.org)

1.基础概念

  1. 生产者消费者
    是一种设计模式，基于一个容器(队列)来解决强耦合的概念

  2. JUC阻塞队列(BlockingQueue)存取方法
    生产者存储数据

消费者获取数据

ArrayBlockingQueue

该队列，初始化的时候必须指定当前队列长度。（定长）

虚假唤醒 阻塞队列中，如果需要线程挂起操作，判断有无数据的位置要采用while循环（不能换成if判断）

默认情况下越小的越靠前。
